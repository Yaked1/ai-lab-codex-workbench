# AI Agent Coding Workbench v0.1.0

This is the first public release of the AI Agent Coding and Prompting Workbench.

## Highlights

- Offline HTML guide site for opening the core learning path in a browser without external scripts, analytics, CDNs, or remote fonts.
- Markdown playbooks for safe agent task intake, branch workflow, prompt design, validation, review, merge, rollback, and public repository hygiene.
- Tool comparison docs for AI coding agents and related workflows, written conservatively so fast-changing product behavior can be verified in official docs.
- Prompt templates for Codex and comparable tools, with scope, inputs, safety boundaries, success criteria, verification steps, final report format, and failure cases.
- Safety model based on explicit scope, local checks, public-safe examples, no secrets, reviewable diffs, human review, and reversible changes.
- Windows-friendly workflow examples using PowerShell, standard-library Python scripts, and GitHub Actions.
- Manual release packaging workflow that creates a GitHub Release ZIP asset and JSON manifest without npm, PyPI, Docker, binary package, or GitHub Packages publishing.

## Package Assets

This release provides:

- `ai-agent-coding-workbench-v0.1.0.zip`
- `package-manifest-v0.1.0.json`

The ZIP contains the public docs, prompt templates, scripts, tests, top-level project files, and GitHub workflow files. The JSON manifest lists packaged files with sizes and SHA-256 hashes, plus the release ZIP hash.

## Intended Audience

This release is meant for maintainers, students, and small teams who want a
reviewable starting point for agent-assisted repository work. It assumes the
reader can open a terminal, read a diff, and run local checks, but it does not
assume deep GitHub Actions, package-management, or model-provider experience.

Use this release when you need:

- A public-safe example of an AI coding-agent workbench.
- Beginner-friendly prompt templates that still include audit fields.
- Windows PowerShell examples for branch, check, package, and review workflows.
- Offline documentation that can be opened without external web assets.
- A small repository that avoids heavy dependencies, local model hosting, and
  GPU-first assumptions.

Do not use this release as a drop-in secret scanner, policy engine, model
gateway, or autonomous publisher. It is educational infrastructure that should
be adapted through normal branch, pull request, and review practices.

## Validation Record

Before publishing a release from this repository, maintainers should record the
exact commands they ran. For this first release line, the expected validation
shape is:

```powershell
python scripts/repo_health_check.py
python scripts/safe_autofix.py --check
python -m unittest discover -s tests
python scripts/build_release_package.py --version v0.1.0
```

The package manifest should be opened and spot-checked after the package build.
Confirm that paths are repository-relative, hashes are present, generated cache
folders are absent, and no private local machine path appears in the JSON.

## Review Notes

Reviewers should pay special attention to:

- Whether examples are public-safe placeholders rather than real account data.
- Whether tool behavior is described conservatively.
- Whether prompt templates contain clear scope, inputs, safety boundaries,
  success criteria, verification steps, final report format, and failure cases.
- Whether automation is framed as human-reviewed and preview-first.
- Whether static HTML remains offline-safe and free of trackers, CDNs,
  analytics, remote fonts, or external JavaScript.

## Known Limitations

- This is a documentation and prompt-template package, not an installable software library.
- The included checks are lightweight guardrails, not a replacement for professional secret scanning or security review.
- Tool behavior, pricing, model availability, and platform support can change quickly and should be verified in official documentation.
- The release workflow is manual and intentionally does not publish on every push.
- The package does not include local private files, generated artifacts, virtual environments, caches, model files, or registry publishing metadata.

## Upgrade Guidance

Future releases should preserve the same safety posture: small diffs, explicit
verification, manifest review, and public repository hygiene. If a future
release adds broader automation, it should document the new permission boundary,
the files that automation may write, and the exact rejection cases that prevent
unreviewed publication.

## Why This Release Shape Was Chosen

The v0.1.0 release intentionally ships as a single ZIP-plus-manifest bundle
rather than an installable package on any registry. The project's actual
artifact is Markdown guides, static HTML, prompt templates, and small
standard-library scripts, not a library other software imports at runtime.
Packaging it as a plain archive keeps the release reviewable end to end: a
maintainer can open the ZIP, diff its contents against the working tree, and
confirm nothing beyond the intended docs and scripts is present, without
reasoning about registry-specific publishing permissions, scoped tokens, or
a separate package manifest format. This choice is recorded here so future
maintainers do not need to reconstruct the reasoning before deciding whether
a registry package is worth adding later.

## Scope Of This Release

v0.1.0 packages the repository state at the time of tagging using the
allowlist enforced by `scripts/build_release_package.py`: the top-level
project files (`README.md`, `AGENTS.md`, `CONTRIBUTING.md`, `SECURITY.md`,
`CHANGELOG.md`, `LICENSE`), the `data/`, `docs/`, `prompts/`, `scripts/`, and
`tests/` directories, and the `.github/workflows/` and
`.github/codex/prompts/` subdirectories. Everything outside that allowlist —
including any local `dist/`, `build/`, `logs/`, `outputs/`, virtual
environment, or cache directory — is excluded by the same script regardless
of what happens to exist on a contributor's machine at build time. This is
what makes the release reproducible: the package contents are a function of
what is committed under the allowlisted paths, not of local build-machine
clutter.

## Reproducing This Release Package Today

The exact commands below rebuild an equivalent package from the current
`main` branch using the real scripts in this repository, so a reader can
verify the mechanics described in this release note rather than take them on
faith. Run these from the repository root in PowerShell:

```powershell
git status
python scripts/repo_health_check.py
python scripts/safe_autofix.py --check
python -m unittest discover -s tests
python scripts/build_release_package.py --version v0.1.0
```

This produces `dist/ai-agent-coding-workbench-v0.1.0.zip` and
`dist/package-manifest-v0.1.0.json`. Rebuilding today will not byte-for-byte
match the original v0.1.0 archive unless the working tree is checked out at
the exact original commit, because the repository has continued to change
since this release: newer guides, additional tests, and updated scripts are
now included by the same allowlist rules. What does stay constant is the
packaging logic itself — the same six top-level files, the same five
top-level directories, the same workflow subdirectories, and the same
exclusion rules for caches, archives, secrets, and oversized files. Anyone
who wants the literal original v0.1.0 bytes should check out the `v0.1.0`
git tag before running the build command, rather than running it against the
current tip of `main`.

To confirm the rebuilt manifest looks reasonable without opening the full
JSON file:

```powershell
Get-Content .\dist\package-manifest-v0.1.0.json -TotalCount 40
(Get-Content .\dist\package-manifest-v0.1.0.json -Raw | ConvertFrom-Json).files.Count
```

alter that factual history. Readers comparing this note against the current

  original v0.1.0 highlights list above.

  and its dedicated release line are separate from the whole-repository
  package described in this note; consult
  [docs/releases-and-packages.md](../releases-and-packages.md) for how the
  two package types relate.
- Additional unit tests may exist under `tests/` beyond what existed at
  v0.1.0 tagging time, which is expected as the test suite grows with the
  repository.
- Later releases (see [docs/releases/release-process.md](release-process.md)
  for the current runbook) may introduce process refinements that did not
  exist when v0.1.0 was cut; where a later process differs from what is
  described here, the release-process document is the current source of
  truth and this file remains a historical record of the first release.
