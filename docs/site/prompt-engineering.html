Prompt design for reviewable work

Give agents enough context to succeed and enough boundaries to stay safe.

A good coding-agent prompt is not just a request. It is a task contract with scope, constraints, examples, verification, and a final report format.

Prompt Pattern

Context

What is the repo?

Name the project goal, audience, active branch, and files likely involved.

Objective

What should change?

Use one clear outcome. Avoid asking for broad cleanup and feature work together.

Boundaries

What must not change?

Exclude secrets, dependencies, workflow YAML, unrelated refactors, and private paths.

Evidence

How will we verify?

List commands, tests, screenshots, or manual review steps that prove the task is done.

Reporting

What should be returned?

Require changed files, commands run, checks run, risks, and claims needing verification.

Recovery

What if it fails?

Tell the agent how to handle unrelated failures, blockers, conflicts, and risky commands.

Reusable Prompt Skeleton

Objective:
Describe the one outcome you want.

Context:
- Repository purpose:
- Audience:
- Current branch:
- Relevant files:

Scope:
- Include:
- Exclude:

Safety:
- Do not add dependencies unless explicitly approved.
- Do not edit secrets, private files, or browser profiles.
- Mark fast-changing tool behavior as "verify in official documentation."

Verification:
- Run:
- Inspect:
- Report anything not verified.

Final response:
- Summary
- Files changed
- Commands run
- Checks run
- Risks and manual verification items

Prompt Engineering Patterns

Pattern Use it when Example instruction
Inspect then edit The agent may not know local conventions. Read `AGENTS.md` and the target files before changing anything.
Smallest correct change The task is focused and reviewability matters. Keep the diff minimal and avoid unrelated refactors.
Verification-first final report You need trustworthy completion status. List exact commands run and distinguish failures from skipped checks.
Claim control The docs mention external products. Do not state pricing, plan access, or model availability without official-doc verification.

Prompt Audit Checklist

Before sending

  • The task has one primary objective.
  • Included and excluded files are clear.
  • The agent knows which checks to run.
  • The prompt forbids secrets and private paths.
  • The final report format is explicit.

Common prompt mistakes

Avoid prompts that ask for "improve everything", skip file inspection, omit checks, request exact fast-changing product claims, or let the agent install dependencies without approval.

Weak Prompt To Strong Prompt

Weak

Make the repo better and add AI docs.

This is too broad. It does not say what to edit, what to avoid, or how to verify success.

Stronger

Read AGENTS.md, README.md, and docs/workflows/public-repo-safety.md.
Add one beginner-friendly section about reviewing AI-generated PRs.
Do not edit workflow YAML, dependencies, or private files.
Run the repo health check and safe autofix check.
Final report: summary, files changed, checks run, and remaining risks.

Failure Modes

Failure Why it happens Prompt fix
Large unrelated diff The task was too broad or the agent reformatted too much. State exact files and forbid unrelated refactors.
Stale product claim The agent guessed current pricing, model access, or setup. Require "verify in official documentation" for fast-changing facts.
Unsafe command The prompt did not constrain install, delete, or system settings commands. List safe commands and require approval for risky commands.
